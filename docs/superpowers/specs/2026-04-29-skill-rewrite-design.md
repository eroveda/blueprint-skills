# Design: Blueprint Skills v0.1 Rewrite — Stack-Agnostic + New Examples

## Problem

Blueprint Skills v0.1 has three critical issues before public launch:

1. **Skills are coupled to Java/Quarkus** — examples, verification commands, class references assume a specific stack
2. **Flagship example is weak** — TaskFlow API is a generic CRUD, incomplete (missing specs/docs), with no verifiable evidence
3. **`universal-framework` produces no structured output** — promises universality but delegates everything to `swebok-decompose`

## Decision

Rewrite all 7 SKILL.md files with a stack-agnostic mindset and replace the example with two new cases that demonstrate real versatility.

## Scope

### Skills — Changes

#### `swebok-decompose`
- Remove hardcoded TaskFlow example
- Replace with a stack-agnostic example (no technology-specific references in output)
- Change verification from `"./mvnw test"` to generic: "run the project's test command"
- Add guidance for multi-layer projects (reference `universal-framework`)

#### `swebok-generate-spec`
- Remove all Java/Quarkus/Maven references from template and example
- `input_context` example must be generic: "project at base package/module X, entity Y exists with fields..."
- `verification` changes to `"run project test suite"`
- Task CRUD example rewritten without FQN, `@TenantContext`, `com.taskflow.*`
- Add **spec ordering/sequencing** section — how to determine execution order from dependency graph

#### `universal-framework`
- Add a **structured output format** (currently produces nothing concrete, just says "use swebok-decompose")
- Add more depth to each of the 8 project types (enough for the AI to apply them correctly)
- Add **"Signals to detect project type"** section — how the AI identifies project type from keywords in user input

#### `generate-architecture-doc`
- Remove Flyway, `@RolesAllowed`, JPA references
- Use generic placeholders: "ORM/data layer", "auth mechanism", "migration tool"

#### `generate-functional-flows`
- Minor review only — already stack-agnostic

#### `generate-user-guide`
- Minor review only — already stack-agnostic

#### `generate-api-catalog`
- Support non-REST interfaces: CLI, SDK, message queues, gRPC
- Add **authentication flows** section — full flow: register, login, refresh, revoke

#### All 7 skills
- Improve frontmatter `description` so the AI knows when to invoke and when NOT to invoke each skill

### Examples — Changes

#### Remove
- `examples/taskflow-api/` — redundant with new flagship, currently empty/incomplete

#### Create: Marketing Automation Platform (flagship)
- **Path**: `examples/marketing-platform/`
- **Reference**: Mailchimp-like
- **Input**: "A marketing automation platform where agencies manage contacts, design email campaigns, set up automations (drip sequences), and track performance (opens, clicks, conversions)."
- **Actors**: Agency Admin, Marketer, Viewer/Client
- **Modules**: Contacts & Lists, Campaign builder, Template engine, Automation workflows, Analytics & tracking, Billing & usage metering, Notifications
- **Demonstrates**: Blueprint decomposes a complex multi-module SaaS with async components (email sending, tracking pixels, automation triggers) into executable specs and professional documentation

#### Create: Fraud Detection Pipeline (second example)
- **Path**: `examples/fraud-detection-pipeline/`
- **Reference**: Stripe Radar-like
- **Input**: "A real-time fraud detection system that ingests financial transactions, extracts features, trains ML models, scores transactions in real-time, and triggers alerts for suspicious activity."
- **Actors**: Data Engineer, Data Scientist, Fraud Analyst, Admin
- **Modules**: Transaction ingestion (streaming), Feature engineering, Model training & evaluation, Real-time scoring service, Alert & case management, Model monitoring & drift detection
- **Demonstrates**: "Universal" is real — Blueprint applies the same methodology to an ML pipeline as to a web SaaS

#### Example structure (both)
```
examples/[name]/
├── README.md              — case study
├── 01-input.md            — original idea as free text
├── 02-framework.md        — universal-framework output
├── 03-decomposition.md    — swebok-decompose output
├── 04-specs/              — specs generated by swebok-generate-spec
│   ├── 01-bootstrap.md
│   ├── 02-data-models.md
│   └── ...
└── 05-documentation/      — docs generated by doc skills
    ├── ARCHITECTURE.md
    ├── FUNCTIONAL_FLOWS.md
    ├── USER_GUIDE.md
    └── API_CATALOG.md
```

No generated code in the repo — examples show Blueprint's output (specs + docs), not the code generated afterwards. Code generation is the responsibility of Claude Code / Superpowers, not Blueprint.

## What does NOT change
- `docs/PHILOSOPHY.md` and `docs/SWEBOK-MAPPING.md` — still valid
- `plugin.json`, `CONTRIBUTING.md`, `LICENSE` — no changes
- `README.md` — updated later during evidence phase of launch plan

## Constraints
- Skills must not overlap with Superpowers (which handles process: brainstorming, TDD, debugging, code review, plan execution)
- Blueprint handles domain: decomposition, specs, documentation
- No ambiguity handling in skills — that's Superpowers' brainstorming flow

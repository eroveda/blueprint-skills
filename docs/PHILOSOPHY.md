# Blueprint Philosophy

## The Problem

Most AI coding tools optimize for "make the prompt work" — not "make the code maintainable in production". The result is code that compiles but lacks structure, tests, documentation, and architectural integrity.

## The Insight

The cheapest bug is the one prevented in the specification. Code generation works best when the spec is precise, complete, and verifiable.

## The Approach

Blueprint Skills enforce structure **before** code generation:

1. **Universal Framework** (INPUTS → PROCESOS → OUTPUTS → SOPORTE) — works for any software type
2. **SWEBOK Categorization** — every component has a clear role (DESIGN, CONSTRUCTION, SECURITY, TESTING, OPERATIONS)
3. **Verifiable Specs** — every spec has acceptance criteria that can be tested
4. **Professional Documentation** — generated alongside code, not after

## Why SWEBOK?

SWEBOK (Software Engineering Body of Knowledge) is the IEEE standard for software engineering practices. Using it as the categorization taxonomy means:

- Specs are aligned with international standards
- Output is recognizable to enterprise/government clients
- Documentation can claim ISO/IEEE compliance
- Knowledge transfers between projects and teams

## Why Universal Framework?

The four-dimension framework (INPUTS → PROCESOS → OUTPUTS → SOPORTE) covers every software project type — from simple REST APIs to IoT devices, blockchain contracts, ML pipelines, and research simulations. One mental model for all decomposition.

## What Blueprint Is NOT

- ❌ A code generator (use Claude Code, Cursor, etc. for that)
- ❌ A workflow tool (use Superpowers for process methodology)
- ❌ A project management tool (use Jira/Linear)
- ❌ A complete framework that handles execution

## What Blueprint IS

- ✅ A spec engine that produces executable specifications
- ✅ A documentation generator aligned with IEEE standards
- ✅ A complement to existing AI coding workflows
